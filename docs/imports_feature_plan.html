1 — Feature Overview

A guided import wizard lets users bulk-upload historical investment data from a CSV file. The system processes the file asynchronously via Sidekiq, auto-resolves or creates related records (Instruments, Follios, Platform Accounts), optionally matches existing bank transactions, and maintains a full audit trail of every import run via a dedicated bridge table.

CapabilityDetail
Import types (Phase 1)Investments active — Transactions, Term Accounts coming soon
Async processingSidekiq job on dedicated imports queue; Redis already available
Progress trackingFrontend polls GET /imports/:id every 1.5 s while status = processing
Import versioningEach upload increments import_version per user+type; starts at v1
Bridge tableimport_records — polymorphic link from batch → any created record
Smart resolutionInstrument (ISIN → ticker → name fuzzy), Platform Account (nickname / platform name), Follio (find_or_create), Transaction matching (amount ± ≤1 + date ± 3 days)
Back-dated datesAny date accepted; no minimum-date validation on import path
Sample CSVDynamic endpoint GET /imports/template/investments — downloads pre-filled example

Smart Resolution Decision Tree

For each CSV row:
─────────────────────────────────────────────────────────────────────────
INSTRUMENT
  1. ISIN present?     → Instrument.find_by(isin: isin)
  2. ticker present?   → Instrument.find_by("LOWER(ticker_symbol) = ?", ticker)
  3. name fallback     → Instrument.where("LOWER(name) LIKE ?", "%name%").first
  4. none found        → Instrument.create!(all columns from row)
  Then: Instruments::TrackService.track(user, instrument) → UserInstrument find_or_create

PLATFORM ACCOUNT  (only if platform_name present in row, else nil)
  1. Match user's PlatformAccounts: nickname (case-insensitive) OR platform.name
  2. Not matched     → find Platform by name (global table)
                         found  → PlatformAccount.create!(user, platform, nickname=platform_name)
                         not found → Platform.find_by(platform_type: :direct) fallback
                                      → PlatformAccount.create!(user, fallback_platform, nickname)

FOLLIO  (only if folio_number present AND user_instrument AND platform_account both resolved)
  → Follio.find_or_create_by(user, user_instrument, platform_account) { f.folio_number = ... }

INVESTMENT
  → Investment.create!(all fields + user_instrument, platform_account)

TRANSACTION MATCH  (best-effort, never blocks row success)
  → user.transactions.active.debit
      .where("ABS(amount - ?) < 1.0", amount_invested)
      .where(date: purchase_date ± 3 days)
      .first
  If found → update_column(:instrument_id, instrument.id)

IMPORT RECORD (bridge)
  → ImportRecord.create!(batch, importable: investment, row_index, status: ok/error, notes)
─────────────────────────────────────────────────────────────────────────

6 — Backend Implementation Plan

P0Sidekiq Setup

  • Add gem "sidekiq", "~> 7.0" and gem "csv" to Gemfile
  • Create config/initializers/sidekiq.rb — connect to Redis using credentials.redis_url
  • Create config/sidekiq.yml — queues: imports(2), default(1)
  • Set config.active_job.queue_adapter = :sidekiq in application.rb
  • Add sidekiq process to Procfile.dev

P1Migrations

  • Migration 1: create_import_batches — all fields listed in §2
  • Migration 2: create_import_records — polymorphic bridge table

P2Models

  • ImportBatch — enum status/type, before_create :set_import_version, progress_pct helper
  • ImportRecord — polymorphic belongs_to :importable, enum status
  • User — add has_many :import_batches
  • Investment — add has_one :import_record, as: :importable
  • ImportBatchSerializer — includes nested import_records array

P3Sidekiq Job — Imports::ProcessInvestmentCsvJob

  • Receives import_batch_id, sets status → processing
  • Parses raw_csv with Ruby stdlib CSV (headers: true, header_converters: :symbol)
  • Sets total_rows, then iterates; rescues per-row to avoid aborting the whole batch
  • Calls Imports::ProcessInvestmentRowService per row, then increment!(:processed_rows)
  • Sets status → completed or failed at end

P4Row Service — Imports::ProcessInvestmentRowService

Runs all smart resolution steps in order: Instrument → UserInstrument → PlatformAccount → Follio → Investment → Transaction match → ImportRecord. Each step is isolated — a failure in transaction matching does not fail the row.

  • Date parser accepts YYYY-MM-DD, DD/MM/YYYY, DD-MM-YYYY
  • Reuses Instruments::TrackService for UserInstrument find_or_create
  • Platform fallback: Platform.find_by(platform_type: :direct)
  • Transaction match tolerance: amount difference < 1.0, date range ± 3 days

P5Template Service — Imports::CsvTemplateService

Returns a CSV string with a header row + two example rows (one stock, one mutual fund). Endpoint: GET /api/v1/imports/template/:import_type responds with Content-Type: text/csv and triggers a file download.

P6Controller — Api::V1::ImportsController

7 — Frontend Implementation Plan

New Types (types/index.ts)

ImportType   = 'investments'
ImportStatus = 'pending' | 'processing' | 'completed' | 'failed'

ImportRowResult { row_index, status: 'ok'|'error'|'skipped', notes }
ImportBatch     { id, import_type, status, file_name, total_rows,
                  processed_rows, failed_rows, import_version,
                  progress_pct, import_records: ImportRowResult[], created_at }
ImportListResponse { items, total, page, page_size }

API Layer (api/imports.ts)

  • listImports(page, pageSize) — paginated GET
  • createImport(importType, file) — multipart FormData POST
  • getImport(id) — single batch GET (for polling)
  • downloadTemplate(importType) — fires window.location.href to trigger browser download

Hook Layer (hooks/useImports.ts)

  • useImports(page) — paginated history
  • useImport(id) — single batch with refetchInterval: data?.status === 'processing' ? 1500 : false
  • useCreateImport() — mutation, invalidates ['imports'] on success

ImportWizard Component (components/imports/ImportWizard.tsx)

A single Sheet (640 px) with internal step state. Sub-components handle each step's content. The context options (platform account override, auto-create flags) are stored in a config state object threaded through the steps. The file is stored in local state until Step 5 triggers createImport.

ImportsPage (pages/ImportsPage.tsx)

Standard page shell — PageHeader with "New Import" button + paginated history table. Completed rows are expandable to show import_records. In-progress rows show a live progress bar (via the batch's progress_pct from the list query, which also refetches every 3 s when any row has status=processing).
