FinTrack — Rails Migration Plan

FastAPI/Python → Ruby on Rails  ·  API-only app  ·  Fresh PostgreSQL schema  ·  Near-zero frontend changes

Pros & Cons

✅ Pros

  • Convention over config — Rails generators cut boilerplate vs FastAPI hand-wiring
  • ActiveRecord — callbacks, scopes, audited gem replace SQLAlchemy event listeners
  • Rake tasks — built into Rails; replaces Typer CLI with no extra dependency
  • Rails console — replaces uv run python -m app.cli console, richer REPL
  • Ecosystem — jwt, rack-cors, kaminari, audited, rswag all maintained and opinionated
  • Migrations — simpler than Alembic; datetime-stamp workaround gone
  • Test ergonomics — RSpec + FactoryBot beats pytest + httpx AsyncClient

⚠️ Cons

  • Full rewrite risk — ~3,000+ lines across 12 services, 12 routers, 13 models
  • PG native enums — resolved: DB recreated; Rails string enums used
  • STI models — resolved: no STI; type columns renamed
  • No async — FastAPI is ASGI/async; Rails synchronous by default
  • API docs — mitigated by rswag; requires writing DSL vs FastAPI zero-effort
  • Context switch — slower feature velocity during transition
  • audited gem — creates own audits table; old audit_logs data lost
  • No user-facing benefit — API surface and frontend stay identical
Recommendation: Proceed. DB drop+recreate removes enum risk. No STI — flat models match Python architecture exactly. One high-risk area: balance-update callback chain — after_create on Transaction must exactly replicate _apply_balance_delta or account balances drift silently. Integration-test this before reports and CLI tasks.

⚠️ Polymorphic Association on Transaction — Manual Handling Required

A Transaction links to either an Account or a TermAccount via linked_account_type ("account" / "term_account") and linked_account_id.

Why not belongs_to :linked_account, polymorphic: true?

  • Rails polymorphic expects the type column to hold the class name ("Account", "TermAccount"). Existing values are lowercase — silent load failures.
  • Rails polymorphic can enforce a FK constraint — but the schema intentionally has no FK here (one FK can't reference two tables).

Solution — manual resolver method: 

# app/models/transaction.rb
def linked_account
  case linked_account_type
  when 'account'      then Account.find(linked_account_id)
  when 'term_account' then TermAccount.find(linked_account_id)
  end
end

The after_create :apply_balance_delta callback calls linked_account — same interface, no schema changes required.

3 Authentication

No Devise — custom JWT to preserve existing endpoint paths exactly.

# lib/json_web_token.rb
module JsonWebToken
  SECRET = Rails.application.credentials.secret_key_base
  EXPIRY = 7.days

  def self.encode(payload)
    JWT.encode(payload.merge(exp: EXPIRY.from_now.to_i), SECRET, 'HS256')
  end

  def self.decode(token)
    JWT.decode(token, SECRET, true, algorithm: 'HS256').first
  rescue JWT::DecodeError
    nil
  end
end
  • app/controllers/concerns/authenticatable.rb — current_user, authenticate_user! before_action (401 on fail)
  • Endpoints preserved: POST /api/v1/auth/login, GET /api/v1/auth/me, PUT /api/v1/auth/me

All 6 PG native enums → Rails string enums (varchar columns, no CREATE TYPE): 

enum :transaction_type, { credit: 'credit', debit: 'debit' }, validate: true
enum :investment_type,  { stock: 'stock', mutual_fund: 'mutual_fund', ... }, validate: true
enum :account_type,     { fd: 'fd', ppf: 'ppf' }, validate: true
Python / SQLAlchemy Rails Migration
ARRAY(Text) for tags t.string :tags, array: true, default: []
UUID for public_id t.uuid :public_id, default: -> { "gen_random_uuid()" }
6 PG native enum types t.string :field, null: false + Rails enum on model
server_default: func.now() t.timestamps or default: -> { "NOW()" }
ondelete: "CASCADE" foreign_key: true, on_delete: :cascade
UniqueConstraint([col1, col2]) add_index :table, [:col1, :col2], unique: true

13 Future: GraphQL (post-migration, additive)

GraphQL was considered but deferred — it requires significant frontend changes (Apollo Client, rewritten queries, new cache strategy) which conflicts with the near-zero frontend constraint.

Best candidates for a future /api/graphql endpoint (REST stays in place):

  • GET /reports/dashboard — aggregates accounts + investments + transactions naturally as a graph query
  • GET /investments with multi-type filters — flexible field selection maps well
  • GET /instruments with search + cursor pagination

Gem: graphql-ruby. Add as a separate phase after Rails migration stabilises, letting frontend migrate incrementally.
