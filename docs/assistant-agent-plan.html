Context

FinTrack is a personal finance tracker. The user wants a new Assistant page where they can:

  1. Chat with an LLM about their own financial data — e.g. "show last week's SIP deposits", "what's my net cash flow this month?"
  2. Upload third-party CSVs (broker exports, bank statements) and have the assistant convert them into FinTrack's existing importable CSV formats (transactions / investments / term_accounts).

Hard constraints:

  • Strict user-data isolation — the assistant must never see another user's data.
  • Confined to financial topics — refuse / redirect off-topic questions.
  • Concise, to-the-point responses.

Locked-in decisions

DecisionChoiceWhy
LLM provider Per-user, configured from the UI. Each user picks Anthropic / OpenAI / Ollama, supplies their own API key (encrypted), and can override model + base URL. If unset → app falls back to local Ollama with friendly error handling. Users own their LLM costs and credentials. No shared/admin key. Ollama gives a zero-config dev fallback.
Write capability Read + generate artefacts only Lower blast radius for v1; user kicks off the actual import on the existing /imports page
Chat structure Single rolling chat per user, persisted; per-message context is session-scoped; user can pin older messages as references Simple storage model, predictable token usage, escape hatch for long-running context

Critical safety invariant. Every tool's constructor takes a User instance. Tool args from the LLM never include user_id. Tools call user.transactions.where(...), never Transaction.where(...). This mirrors the existing pattern in app/controllers/api/v1/transactions_controller.rb.

System prompt (sketch)

You are FinTrack's financial assistant for {user.full_name}.
Currency: {user.currency_code}. Today: {Date.current.iso8601}.

You can:
- Answer questions about THIS user's transactions, accounts, term accounts (FD/PPF),
  investments, and platform accounts. Always use the provided tools — never invent figures.
- Convert uploaded CSVs into FinTrack's import format. Use analyse_csv first to inspect
  headers, propose a column mapping, wait for confirmation, then call generate_import_csv.
  Tell the user to open the Imports page to load the generated file.

You must:
- Decline non-financial topics politely and steer back to finance/data/imports.
- Keep responses concise. Tables for lists. Use the user's currency code in numbers.
- Never claim to have edited or imported anything — v1 is read-only + artefact generation.

Provider resolution & fallback

On every POST /messages the orchestrator resolves the provider:

  1. Read current_user.assistant_setting. If provider is set and credentials look valid (key present for Anthropic / OpenAI; base_url reachable for Ollama), use it.
  2. If provider is unset (user hasn't configured) → fall back to local Ollama at http://localhost:11434, model llama3.
  3. If the chosen provider raises a network / auth error mid-call, the controller returns a structured error event over the SSE stream: 
    event: error
data: {
  "code": "provider_unreachable",
  "provider": "ollama",
  "fallback_used": true | false,
  "message": "Ollama is not running on http://localhost:11434. Start it or configure a provider above."
}
    The frontend renders this inline in the chat as an error bubble with a "Open settings" button.
One controlled cascade only. If the user-configured provider fails (e.g. invalid key), we surface the error — we do not silently retry on Ollama. Auto-fallback only applies when the user has never configured a provider, so the user is never surprised by an unexpected provider being used with their data.

Streaming

  • Api::V1::Assistant::MessagesController includes ActionController::Live.
  • POST /messages opens an SSE stream. Events: token, tool_call, tool_result, complete.
  • Sidekiq is not used here — assistant calls are synchronous request-time, streamed back. Tools run inline.

Rake tasks (admin / emergency only)

Settings are primarily UI-driven — these tasks exist for headless setup and support work, not as the primary path.

CommandPurpose
bin/rails "assistant:configure[user@email.com]" Interactive prompts (provider, model, API key, base_url) targeted at the named user's settings row. Validates with a test call before saving.
bin/rails "assistant:status[user@email.com]" Prints that user's configured provider/model and result of a ping.

Rate limiting

One rack-attack throttle: assistant_messages_per_user = settings.daily_limit per 24 h, keyed by current_user.id. Configured in config/initializers/rack_attack.rb.

Audit & telemetry

No mutations on financial data in v1, so the existing audited gem isn't needed for assistant tables. We do log provider, model, tokens_in, tokens_out, latency_ms on every assistant message — added now to avoid a second migration later.

Settings panel behaviour

  • Provider dropdown changes which fields are required:
    • Anthropic: API key required, model defaults to claude-sonnet-4-6, base_url defaults to https://api.anthropic.com.
    • OpenAI: API key required, model defaults to gpt-4o-mini, base_url defaults to https://api.openai.com/v1.
    • Ollama: API key not required, model defaults to llama3, base_url defaults to http://localhost:11434.
  • API key field is masked. The current value is shown only as "…7f2a" (last 4 chars). Clicking Replace reveals an empty input.
  • Test connection calls POST /api/v1/assistant/setting/test with the form values without saving. Backend issues a tiny test call to the chosen provider, returns { ok, latency_ms, error? }. UI shows green / yellow / red indicator.
  • Save persists. The panel collapses to a one-line banner showing provider · model · status dot.
  • If a chat message fails because of a provider error, the panel auto-expands and highlights the offending field.

API & hooks

  • getSetting() → { provider, model, base_url, daily_limit, has_api_key, api_key_tail, last_tested_at, last_test_status }. API key itself is never returned.
  • updateSetting(payload) → PATCH; api_key only sent when user is replacing it.
  • testSetting(payload) → POST; returns { ok, latency_ms, error? } without persisting.
  • listMessages(cursor?) — paginated history (cursor-based, newest first).
  • uploadAttachment(file) → { attachment_id }.
  • streamMessage({ content, attachment_id?, reference_ids? }) — uses fetch + ReadableStream reader (not native EventSource: we need POST body + auth header). Yields token, tool_call, tool_result, error, complete. Appends to a "draft" assistant message in React state; persists via React Query cache update on complete.
  • pinMessage(id) / unpinMessage(id).
  • newSession() — returns { session_id } and updates client state.
  • resetHistory() — DELETE.

Hook: useAssistantSettings()

Returns { data, isLoading, mutate, test, isConfigured }. isConfigured drives whether the panel renders collapsed or expanded. mutate handles save (with optimistic UI). test handles the dry-run check before save.

Reused existing utilities

  • Auth pattern — Authenticatable concern already injects current_user into all Api::V1::* controllers. New Api::V1::Assistant::* controllers inherit it for free.
  • Query services — tools wrap existing Transactions::QueryService, Investments::QueryService, Reports::Dashboard, Reports::SpendingTrends instead of duplicating SQL.
  • CSV reading — same CSV.foreach pattern used in app/jobs/imports/process_*_csv_job.rb.
  • CSV templates — app/services/imports/csv_template_service.rb defines the canonical column lists for each import type. generate_import_csv reads its target headers from this service so the generated files are guaranteed import-compatible.
  • ActiveStorage — already wired post-migration 20260506090521. New attachments use has_one_attached :file on AssistantMessage.
  • Frontend API client — frontend/src/api/client.ts already attaches Bearer token; new assistant.ts uses it directly.
  • Currency / formatting — useCurrency() in messages and tool result rendering.

Out of scope for v1

  • Multiple named conversations (left sidebar). User chose single rolling chat.
  • Write actions (correcting / deactivating txns, dispatching imports). User chose read+generate-only.
  • Cost dashboard (token spend per provider). Columns added now to make it trivial later.
  • Vector search / embeddings of historical transactions. Sliding-window history is sufficient for v1.
  • Voice input / image attachments.
